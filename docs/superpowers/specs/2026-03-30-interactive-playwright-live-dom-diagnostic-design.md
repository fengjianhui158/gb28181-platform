# 摄像机国标配置交互式 Playwright live DOM 代理设计

> 版本: 1.0
> 日期: 2026-03-30
> 状态: 待评审

---

## 1. 背景

当前浏览器诊断主路径已经验证出两个事实：

1. 单靠静态 HTML 导出再做规则提取，不足以稳定处理摄像机页面的动态值和复杂布局。
2. 当前导航失败的核心原因不是“字段值读不到”，而是“没有像人一样点到正确的菜单项”。

以大华设备为例，程序虽然已经具备：

- 登录成功
- 根据厂商配置解析导航路径
- 读取 live DOM 值
- 提取页面可见字段

但自动导航仍可能误点到：

- 隐藏元素
- 祖先容器
- 文本包含目标词但并非菜单项的节点

而在人工排障和 Codex 成功读取配置的链路中，真正可靠的方式是：

- 先看当前页面有哪些可见菜单/页签/按钮
- 点击最像真实菜单项的那个元素
- 每一步都验证页面状态是否变化
- 到达目标页后直接读取 live DOM 控件值

因此本次设计将浏览器诊断主路径正式调整为：

`交互式 Playwright live DOM 代理`

静态 HTML 规则提取降级为辅助手段，截图/多模态模型继续保留为兜底。

---

## 2. 目标

### 2.1 功能目标

- 主路径按“人类工程师网页登录排障”的方式工作。
- 优先解决导航误点和页面状态判断问题。
- 到目标页后，直接基于 live DOM 提取国标 28181 配置值。
- 主框架兼容大华、海康、宇视三类设备。
- 厂商差异尽量通过配置驱动，而不是为每个厂家写一套流程。

### 2.2 非目标

- 不追求一次性覆盖所有厂商所有固件版本的全部页面差异。
- 不让大模型参与主链路导航和主链路字段提取。
- 不删除现有可见字段提取器和 RPC2/截图链路，只调整主次关系。

---

## 3. 核心思路

主路径拆成三个明确层次：

1. 交互式导航代理
2. 目标页识别
3. live DOM 字段提取

这三层都运行在 Playwright 页面上下文之上，不再以“导出 HTML 再离线分析”为核心模式。

---

## 4. 方案对比

### 4.1 方案 A: 继续增强静态 HTML + 可见字段提取

优点：

- 复用现有代码较多
- 单元测试容易写

缺点：

- 动态值、复杂控件、祖先容器误匹配问题仍然反复出现
- 导航问题本质上没有解决
- 与人工排障路径不一致

### 4.2 方案 B: 主路径改为交互式 Playwright live DOM 代理

优点：

- 与 Codex 成功链路一致
- 与交付工程师手工排障一致
- 对动态页面和旧式摄像机前端更稳
- 更容易解释“为什么点了这个菜单、为什么认定当前页是目标页”

缺点：

- 实现复杂度更高
- 集成测试比纯 HTML 规则更重要
- 单次诊断耗时会略有增加

### 4.3 结论

采用方案 B 作为新主路径。

---

## 5. 架构设计

### 5.1 交互式导航代理

职责：

- 根据厂商配置的导航路径候选逐步导航
- 每一步只在当前可见元素中选择候选
- 点击后验证页面状态是否真的变化
- 若当前候选无效，回退并尝试下一个候选

每一步的判定逻辑：

1. 收集当前页面和 iframe 中的可见候选元素
2. 优先候选类型：
   - `button`
   - `a`
   - `li`
   - `tab`
   - `menu item`
   - 语义上最接近菜单项的 `span/td/div`
3. 优先文本精确匹配
4. 再允许包含匹配
5. 点击前滚动到可见区域
6. 点击后检查页面状态变化

页面状态变化的判据可以包括：

- URL 变化
- DOM 文本变化
- 菜单展开状态变化
- 新内容区域出现
- 目标下一步菜单变为可见

如果点击后无变化，则认为该候选无效，继续尝试同一步的其他候选。

### 5.2 目标页识别

不再依赖页面标题。

识别规则：

- 在当前主页面和 iframe 中扫描 live DOM 的可见文本
- 根据 `VisibleFieldAliases` 统计标准字段命中数
- 当命中数达到阈值时，认定当前页是有效的国标配置页

建议阈值：

- `>= 3` 个标准字段命中：认定为候选国标页
- `>= 5` 个标准字段命中：高置信度国标页

### 5.3 live DOM 字段提取

字段提取必须直接读当前控件状态，而不是依赖静态 HTML 字符串。

提取策略：

1. 先根据可见标签文本匹配标准字段
2. 以标签为锚点寻找最近的控件或控件容器
3. 按控件类型读取当前值：
   - `input[type=text|number|password]`
   - `select`
   - `checkbox`
   - 容器中的多输入框
   - 分段 IP
4. 提取结果统一映射到标准字段模型

支持的通用结构模式：

- `label + input`
- `label + select`
- `label + checkbox`
- `label + 容器(div/span) + input...`
- `label + 分段 IP 容器`
- `label/title 属性 + 邻近控件`

### 5.4 兜底层

主路径失败后再进入：

1. 现有 RPC2 读取
2. 截图 + 多模态模型解释

顺序保持“规则与浏览器优先，模型兜底”。

## 6. 配置设计

### 6.1 厂商导航配置

继续沿用 `Diagnostic:Manufacturers:<vendor>:NavigationPaths`

但语义变化为：

- 每一条路径是一组“人工可见菜单文本步骤”
- 程序按步骤执行交互式导航
- 不再假设“文本匹配到任意节点即可点击成功”

示例：

```json
{
  "Diagnostic": {
    "Manufacturers": {
      "dahua": {
        "NavigationPaths": [
          [ "设置", "网络设置", "平台接入", "国标28181" ],
          [ "设置", "网络", "平台接入", "国标28181" ]
        ]
      }
    }
  }
}
```

### 6.2 字段别名配置

继续沿用 `Diagnostic:VisibleFieldAliases`

示例：

```json
{
  "SipServerIp": [
    "SIP服务器地址",
    "SIP服务器IP",
    "SIP Server IP",
    "Server IP"
  ]
}
```

### 6.3 可选后续扩展

后续可以再加但本轮不强制：

- 厂商级候选元素优先级
- 厂商级点击后等待策略
- 厂商级控件结构提示

当前优先保持主引擎通用。

## 7. 数据模型

浏览器采集结果统一输出为：

```csharp
public class VisibleGbConfig
{
    public bool? Enable { get; set; }
    public string SipServerId { get; set; } = "";
    public string SipDomain { get; set; } = "";
    public string SipServerIp { get; set; } = "";
    public string SipServerPort { get; set; } = "";
    public string LocalSipPort { get; set; } = "";
    public string RegisterExpiry { get; set; } = "";
    public string Heartbeat { get; set; } = "";
    public string HeartbeatTimeoutCount { get; set; } = "";
    public string DeviceId { get; set; } = "";
    public string ChannelId { get; set; } = "";
    public Dictionary<string, string> RawMatches { get; set; } = new();
}
```

`RawMatches` 用于记录页面实际命中的标签和值，便于日志与回归验证。

## 8. 日志与可观测性

必须记录以下证据链：

- 当前厂商与导航路径候选
- 每一步找到的候选元素数量
- 最终选择的元素类型、文本、可见性、所在 frame
- 点击后页面状态是否变化
- 当前页命中的标准字段数量
- 每个字段命中的原始标签和值
- 失败原因：
  - 未找到可见菜单项
  - 点击后页面未变化
  - 未命中国标页
  - 字段命中但取值失败

这些日志要足以回答：

- 程序为什么点了这个元素
- 为什么认为它不是目标页
- 为什么提取失败

## 9. 测试策略

### 9.1 单元测试

覆盖：

- 候选元素排序
- 文本精确匹配优先
- 可见元素优先
- 标签到字段的别名匹配
- 分段 IP 与容器控件取值

### 9.2 浏览器级集成测试

新增以 HTML 片段或本地测试页模拟：

- 大华顶部页签 + 左侧菜单
- 海康层级菜单
- 宇视页签/菜单组合

重点验证：

- 交互式导航确实点到正确菜单
- 点击后状态变化判断有效
- 到目标页后能提取 live DOM 值

### 9.3 真机验证

首轮真机验证目标：

- 大华设备
- 成功进入 `设置 -> 网络设置 -> 平台接入 -> 国标28181`
- 成功读到页面上人工可见的 SIP 相关配置

海康、宇视后续用相同框架验证，只优先补配置与少量通用规则。

## 10. 风险与应对

风险 1：旧设备页面结构混乱，菜单元素没有清晰语义标签  
应对：保留多级候选类型和点击后状态校验，不只看单一 selector。

风险 2：同名文本在多个位置重复出现  
应对：可见性优先、精确文本优先、最短文本优先、点击反馈校验。

风险 3：iframe 或异步加载导致菜单查找不稳定  
应对：主页面与 frame 全部纳入候选搜索，点击后增加状态等待和重评估。

风险 4：字段值来自动态控件状态  
应对：主路径直接读取 live DOM 控件值，不再依赖静态 HTML。

---

## 11. 结论

新的浏览器诊断主路径应正式调整为：

`交互式 Playwright live DOM 代理`

这条路线最接近：

- Codex 本次成功获取国标配置的实际操作方式
- 交付工程师网页登录人工排障的方式

也最有希望作为一个可以扩展到大华、海康、宇视三厂家的通用浏览器采集框架。
