# 摄像机国标配置可见字段采集设计

> 版本: 1.0  
> 日期: 2026-03-30  
> 状态: 待审阅

---

## 1. 背景

当前浏览器诊断流程对大华设备优先依赖 RPC2、页面内部变量和大模型分析。这条链路存在三个问题：

1. 不同厂商、不同型号、不同固件的内部变量名不稳定。
2. RPC2 配置名和返回结构不统一，容易出现“页面上能看到，程序里拿不到”。
3. 交付工程师实际排障时依赖的是网页登录后肉眼可见的配置页面，程序采集路径应尽量与人工排障路径一致。

本次设计将浏览器诊断的主路径改为“页面可见字段采集”，即程序像人工排障一样登录网页、进入国标配置页面、读取页面上可见的业务标签和值，再统一映射为标准字段模型。

---

## 2. 目标

### 2.1 功能目标

- 支持大华、海康、宇视等厂商的网页国标配置采集。
- 主路径基于页面上肉眼可见的标签文本和值，不依赖页面标题、DOM 内部变量名或特定 RPC2 接口。
- 将不同文案归一为统一标准字段模型。
- 后续新增厂商或型号时，优先通过 `appsettings.json` 配置扩展，不要求频繁修改 C# 代码。

### 2.2 非目标

- 不追求一次性覆盖所有品牌全部页面差异。
- 不把大模型作为主采集链路。
- 不删除现有 RPC2 与 AI 分析能力，只将其降级为兜底。

---

## 3. 设计原则

### 3.1 主路径只看页面可见信息

主路径中的字段识别对象是用户在浏览器页面上能直接看到的标签文本，例如：

- `SIP服务器地址`
- `SIP服务器IP`
- `SIP Server IP`
- `SIP域`
- `接入使能`

程序不以 DOM 内部 `id`、`name`、`class` 作为主识别依据。这些内部属性只可作为辅助取值线索。

### 3.2 标准字段统一建模

系统内部只使用统一标准字段名，页面文案通过别名归一。例如：

- `SipServerIp`
- `SipServerId`
- `SipDomain`
- `SipServerPort`
- `LocalSipPort`
- `Heartbeat`

### 3.3 配置驱动优先

厂商差异、菜单差异、字段同义词差异优先通过配置处理。代码负责通用流程、归一化、定位和日志。

### 3.4 LLM 只做兜底

大模型仅在规则无法可靠识别字段时参与解释，不参与主路径导航和主路径采集。

---

## 4. 方案对比

### 4.1 方案 A：配置驱动的可见字段采集

核心思路：

- 使用配置定义厂商导航候选路径。
- 使用配置定义标准字段的可见标签别名。
- 到达候选页面后，扫描页面可见文本并按别名匹配字段。
- 在标签附近查找实际控件值并组装结果。

优点：

- 符合“主要改配置即可支持新型号”的要求。
- 逻辑可测试、可解释、可回归。
- 与交付工程师实际操作路径一致。

缺点：

- 首次需要建立字段别名配置和标签邻近取值规则。

### 4.2 方案 B：按厂商/型号写死 CSS 或 ID

优点：

- 单一型号上稳定。

缺点：

- 不同型号很容易失效。
- 维护成本高。
- 与“主要改配置支持”目标冲突。

### 4.3 方案 C：LLM 主导识别 DOM 或截图

优点：

- 对文案变化有一定容忍度。

缺点：

- 成本高、稳定性差、难以自动化验证。
- 不适合作为生产主路径。

### 4.4 结论

采用方案 A 作为正式方案，RPC2 与 LLM 只保留为兜底。

## 5. 配置设计

配置分为两层：

1. 全局标准字段别名配置
2. 厂商导航候选路径配置

### 5.1 全局字段别名配置

建议新增：


- 后续新增型号时优先补充 aliases，而不是改采集代码。

- 后续如遇新菜单结构，优先补配置。

## 6. 标准字段模型

建议在浏览器采集层统一输出如下字段模型：

```csharp
public class VisibleGbConfig
{
    public bool? Enable { get; set; }
    public string SipServerId { get; set; } = "";
    public string SipDomain { get; set; } = "";
    public string SipServerIp { get; set; } = "";
    public string SipServerPort { get; set; } = "";
    public string LocalSipPort { get; set; } = "";
    public string RegisterExpiry { get; set; } = "";
    public string Heartbeat { get; set; } = "";
    public string HeartbeatTimeoutCount { get; set; } = "";
    public string DeviceId { get; set; } = "";
    public string ChannelId { get; set; } = "";
    public Dictionary<string, string> RawMatches { get; set; } = new();
}
```

说明：

- 字段模型内部使用统一标准字段名。
- `RawMatches` 用于记录页面原始标签和值，便于日志和调试。
- 密码字段允许只记录“存在且已配置/掩码显示”，不要求明文输出。

## 7. 运行流程

### 7.1 主流程

1. 启动 Playwright。
2. 登录摄像机网页。
3. 根据厂商配置依次尝试候选导航路径。
4. 每进入一个候选页面，扫描页面可见文本。
5. 根据 `VisibleFieldAliases` 判断当前页面是否命中国标配置字段。
6. 如果命中足够多的标准字段，则围绕标签提取控件值。
7. 组装为 `VisibleGbConfig`。
8. 与平台期望配置对比，输出诊断结论。
9. 若主路径失败，再进入 RPC2/LLM 兜底。

### 7.2 页面识别规则

页面是否被判定为“有效国标配置页”，不依赖页面标题，而依赖标准字段命中数。

建议规则：

- 命中标准字段数 `>= 3`：认为页面大概率为有效配置页
- 命中标准字段数 `< 3`：认为不是目标页，继续尝试下一条导航路径

### 7.3 可见标签匹配规则

标签匹配步骤：

1. 提取页面当前可见文本节点。
2. 对文本做归一化处理：
   - 去除首尾空格
   - 多空格合并
   - 全角半角统一
   - 中英文大小写忽略
   - 去掉末尾冒号、空格等装饰字符
3. 与标准字段 aliases 做等值或近似匹配。

### 7.4 取值规则

定位到标签后，按以下优先级取值：

1. 标签附近的 `input`
2. 标签附近的 `select`
3. 标签附近的 `checkbox` / `radio`
4. 标签同容器中的只读文本节点
5. 标签附近的密码框掩码值
6. DOM 内部 `id/name` 仅作为辅助

特殊规则：

- IP 被拆成 4 段输入框时，需要按顺序拼接成完整 IP。
- `checkbox` 或 `radio` 需要映射为 `bool?`。
- `select` 读取当前选中文本，不只读取内部 value。

## 8. 代码改造边界

### 8.1 保留不变的入口

- `BrowserCheckStep` 继续作为诊断步骤入口。

### 8.2 调整 `CameraBrowserAgent` 职责

`CameraBrowserAgent` 从“AI 导航与多来源混合分析器”调整为“浏览器采集协调器”，主要负责：

- 登录
- 导航
- 调用可见字段采集器
- 组织结果和日志
- 在主路径失败时调用兜底路径

### 8.3 新增可见字段采集器

建议新增独立组件，例如：

- `VisibleFieldConfigExtractor`
- `VisibleFieldAliasOptions`
- `ManufacturerNavigationOptions`

职责：

- 读取配置
- 归一化页面文本
- 匹配 aliases
- 提取标签附近控件值
- 输出统一字段模型

### 8.4 现有能力调整

- `DahuaRpc2Client` 保留，但只作为兜底，不再做主路径。
- Qwen 文本分析保留，但仅用于解释结果或兜底识别，不参与主采集链路。
- 截图能力保留，用于失败诊断证据，不作为默认采集方式。

---

## 9. 失败判定与兜底策略

### 9.1 失败判定

以下情况视为主路径失败：

- 所有导航候选路径都无法进入有效配置页。
- 页面标准字段命中数不足阈值。
- 字段标签命中但附近控件取值失败。
- 取值结果严重缺失，无法形成有效配置模型。

### 9.2 兜底顺序

主路径失败后，按顺序进入：

1. RPC2 配置拉取
2. DOM 辅助内部属性提取
3. LLM 文本解释
4. 截图 + 视觉模型分析

说明：

- 兜底链路保留现有能力价值，但只作为辅助，不主导流程。
- 即使进入兜底，也必须在日志中明确标记“主路径失败原因”。

---

## 10. 日志与可观测性

必须记录以下证据：

- 当前厂商识别值
- 当前尝试的导航路径
- 每一步成功点击的菜单文本
- 当前页面命中的标准字段及其 alias
- 每个字段的取值来源
  - 标签邻近 input
  - 标签邻近 select
  - checkbox/radio
  - 只读文本
  - DOM 辅助
  - RPC2 兜底
  - LLM 兜底
- 主路径失败原因
- 最终诊断结论

目标是让工程师从日志中直接看到：

- 程序点了哪里
- 识别到了什么标签
- 为什么拿到了或没拿到某个字段

## 11. 测试策略

### 11.1 单元测试

新增测试覆盖：

- 文本归一化
- aliases 匹配
- 标签附近控件选择
- IP 四段拼接
- checkbox/select/password 的值提取

### 11.2 集成测试

使用固定 HTML 片段模拟多厂商页面：

- 大华页面
- 海康页面
- 宇视页面

每个 HTML 片段都应验证：

- 是否正确命中国标配置页
- 是否能提取出统一标准字段
- 是否能正确处理字段文案差异

### 11.3 回归测试目标

后续新增型号时，优先通过：

1. 补充别名配置
2. 补充导航路径配置
3. 增加对应 HTML 样例测试

而不是直接修改主流程代码。

## 12. 风险与约束

### 12.1 风险

- 同一页面标签与控件布局差异过大，邻近取值规则可能需要补强。
- 个别厂商页面存在 iframe、多层异步渲染或自绘组件，需要额外适配。
- 某些页面标签过于简略，仅靠别名可能不够，需要上下文联合判断。

### 12.2 控制措施

- 优先通过配置补充字段 aliases 和导航路径。
- 采集器必须支持遍历主页面和 iframe。
- 日志中保留原始匹配证据，便于后续快速补配置。
- 保留 RPC2/LLM/截图兜底能力，避免完全失效。

---

## 13. 实施结果预期

改造完成后，浏览器诊断应达到以下效果：

- 对大华摄像机，能够像人工登录网页一样稳定读取国标配置页可见值。
- 对海康、宇视摄像机，主要通过补充配置支持新菜单和新文案。
- 当字段文案从 `SIP服务器地址` 变成 `SIP服务器IP` 或 `SIP Server IP` 时，不需要改主代码。
- 诊断日志能清晰说明识别过程和失败原因。

---

## 14. 结论

本方案将浏览器诊断主路径从“内部接口/模型猜测”调整为“页面可见字段采集”，更接近工程师真实排障方式，也更适合多厂商、多型号的长期维护。

正式实施时应坚持以下边界：

- 主路径：配置驱动的可见字段采集
- 兜底：RPC2、DOM 内部属性、LLM、截图
- 扩展方式：优先补配置和测试，而不是堆叠厂商硬编码
